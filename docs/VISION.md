# Подсистема Vision: режимы и настройка

## Режимы
- `VISION_MODE=dummy` - полностью автономный стендовый режим без камеры: `people_count` и список распознанных ID берутся из ENV (`VISION_DUMMY_PEOPLE`, `VISION_DUMMY_RECOGNIZED`), а видеопоток формируется как стендовый MJPEG-поток. Удобно для автономной проверки и настройки интерфейса.
- `VISION_MODE=real` - работа с камерой + OpenCV YuNet + SFace (если модели доступны) либо Haar + запасной дескриптор. MJPEG формируется на стороне backend; браузер не получает доступ к камере напрямую.

Если моделей нет и `VISION_AUTO_DOWNLOAD=1`, backend автоматически скачает YuNet/SFace в `./models/`.

## MJPEG и оверлеи
- Эндпоинт: `GET /api/video/mjpeg` (долгоживущий, работает в нескольких вкладках). Страница `/monitor` пытается переподключиться через кнопку "Reload stream".
- В `/ws/status` (или `/api/status/`) передаются:
  - `people_count` (счетчик силуэтов на основе вычитания фона)
  - `silhouettes[*].x/y/w/h` (bbox силуэтов для отрисовки)
  - `faces[*].box` + `frame_w/frame_h` (масштабирование координат)
  - `faces[*].label` (имя/логин) или `UNKNOWN` + флаг `is_known`
  - `vision_state`, `vision_error`, `last_frame_ts`, `fps`
- При отказе камеры/vision выставляется `vision_error`, UI показывает `CAMERA DOWN`, Door2 остается заблокированной.

## Настройка (ENV)
- Детекция/сопоставление: `VISION_DET_SCORE` (по умолчанию 0.7), `VISION_MATCH_THRESHOLD` (порог), `VISION_MATCH_METRIC` (`l2`/`cosine`).
- Источник камеры: `VISION_CAMERA_INDEX` (0/1/2...) для выбора `/dev/videoN` (Linux) или индекса камеры (Windows).
- Силуэты: `BG_HISTORY`, `BG_VAR_THRESHOLD`, `PEOPLE_MIN_AREA`, `VISION_TTL_SEC` (устаревание кадра), `ROOM_CHECK_SAMPLES` (агрегация в FSM).
- Устаревание кадров: `VISION_TTL_SEC` задает время валидности кадра; при устаревании Door2 блокируется и UI показывает `CAMERA DOWN`.

### Если видно "слишком много силуэтов"
Обычно это значит, что вычитание фона слишком чувствительно к освещению/шуму камеры.

Попробовать (увеличивать постепенно):
- `BG_VAR_THRESHOLD=40` (меньше чувствительность)
- `PEOPLE_MIN_AREA=8000` (фильтрация мелких пятен; для 1080p может потребоваться `15000+`)

### Фиксированная метка для стендовой проверки
Если нужно, чтобы UI стабильно показывал **одно "известное" имя** в автономном стендовом режиме, можно задать:
- `VISION_DEMO_FIXED_LOGIN=shapovalov` (пользователь должен существовать в БД; при `EYEGATE_DEMO_MODE=1` он создается сидом)

## Замечания по безопасности
- Счетчик людей основан на силуэтах; при `people_count==0`, но наличии лиц, UI все равно рисует боксы, однако политика может требовать `people_count==1` для открытия Door2.
- При `VISION_MODE=real` и устаревании кадров FSM считает vision недоступным (Door2 заблокирована; возможна тревога при >1 человеке).

## Камера в WSL (частая проблема)
Если backend запущен внутри WSL и камера не проброшена, OpenCV не откроет `/dev/video*`, UI покажет `CAMERA DOWN` (Door2 остается заблокированной политикой).

### Исправление (Windows 11 + WSL2 через usbipd)
1) На Windows (PowerShell от имени администратора):
   - `usbipd list`
   - `usbipd bind --busid <BUSID>`
   - `usbipd attach --wsl --busid <BUSID>`
2) Внутри WSL:
   - `ls -l /dev/video*`
   - убедиться, что текущий пользователь имеет доступ (например, добавить в группу `video` при необходимости).
3) Перезапустить backend и проверить, что `/monitor` запрашивает `/api/video/mjpeg`, а в статусе `vision.camera_ok=true`.

### Если `/dev/video*` есть, но кадры не приходят
Некоторые встроенные веб-камеры работают только в isochronous-режиме. WSL USB passthrough может показать устройство, но поток кадров не пойдет (OpenCV `cap.read()` тайм-аутится; `dmesg` может спамить `usb usb1: Not yet implemented`).

В этом случае:
- запустить backend на хосте (Windows/Linux), где камера доступна напрямую, или
- использовать другую USB-камеру, которая работает в bulk-режиме.

### Альтернатива
- Запустить backend на хостовой ОС (Windows/Linux), где камера доступна напрямую.
