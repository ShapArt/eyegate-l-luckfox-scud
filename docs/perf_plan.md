# План нагрузочных тестов и устойчивости

## Цели
- Проверить, что FSM/GateController устойчивы при потоке событий и таймеров.
- Оценить отклик API (FastAPI) при параллельных запросах login/status/events.
- Проверить рост таблицы events и влияние на время выборки.
- Убедиться, что event-loop не блокируется, таймеры срабатывают в срок.

## Сценарии (описательно)
1. **Burst AuthResult/door events (FSM):**
   - Сгенерировать N (например, 1000) последовательностей: AuthResult(allow=True) → DoorClosedChanged(1, True) → RoomAnalyzed(1, MATCH) → DoorClosedChanged(2, True).
   - Проверить отсутствие зависаний, конечное состояние RESET, число ALARM/ACCESS_DENIED = 0.
   - Метрики: среднее время обработки цикла, число непроцессed events в очереди.

2. **Parallel login/status/events (API):**
   - httpx/asyncio: M параллельных login запросов (валидные/невалидные) + периодические GET /api/status и /api/events.
   - Проверить, что 5xx отсутствуют, 429 срабатывает по rate limit, средняя латентность в допустимых пределах.

3. **Рост таблицы events:**
   - Предзаписать 50k событий (скриптом) → GET /api/events?limit=100, измерить время ответа.
   - Проверить, что лимит и сортировка по id DESC работают; БД не падает.

4. **Таймеры под нагрузкой:**
   - Запустить GateController, параллельно генерировать мелкие задержки/бурсты login_success + door events.
   - Проверить, что ENTER/CHECK/EXIT таймауты всё равно срабатывают (с допуском) — не “зависают” из-за перегрузки event-loop.

5. **Vision real (нагрузка):**
   - При включённом OpenCV, прогнать N вызовов analyze_room с тестовыми кадрами; измерить время на один вызов.
   - При перегреве/ошибках — убедиться, что Vision исключения отрабатываются (FSM получает NO_FACE fallback).

## Инструменты
- httpx + asyncio для параллельных API вызовов.
- Простые скрипты/pytest-mark для бурстов GateController (без реального I/O).
- Метрики можно логировать в stdout/файл (простейшая статистика).

## Критерии приемки (базовые)
- Нет 5xx на API при нагрузке; rate limit даёт 429 при brute-force.
- FSM не застревает, очередь событий пустеет, состояние возвращается в RESET/IDLE.
- Таймеры отрабатывают в пределах разумного допуска (плюс 10–20% под нагрузкой).
- Выборка событий с большим количеством строк отвечает < 1с на limit=100.
