# Роли и ACL (предложение)

## Цели
- Разделить права: admin vs user (оператор).
- Защитить CRUD пользователей/экспорт/логов, оставить статус/логин публичными (или ограничить).

## Минимальный вариант (реализовано частично)
- `ADMIN_TOKEN` + заголовок `X-Admin-Token` для /api/users, /api/events, /api/events/export (уже есть).
- Статус/логин/регистрация публичны.

## Полноценный вариант (предложение)
- Таблица `users` дополняется колонкой `role` (enum: admin, operator, user; по умолчанию user).
- JWT/сессии:
  - `/api/auth/login` выдает токен (JWT или session cookie) с role.
  - Все защищённые маршруты проверяют role через Depends.
- ACL:
  - admin: CRUD users, export events, блокировки.
  - operator: read-only events, status, список пользователей (без редактирования), trigger reset.
  - user: только свой профиль/статус (в контексте демо можно ограничиться login/status).

## Изменения в коде (эскиз)
- DB: `role TEXT NOT NULL DEFAULT 'user'` + миграция.
- Pydantic схемы: добавить поле role (ограниченный список).
- Auth: при логине выдавать токен (в проекте — простой HMAC-подпись в `auth/tokens.py`, поля user_id/role/exp).
- Декораторы/Depends для проверки роли (e.g. `require_role('admin')`).
- Web: форма логина сохраняет токен в localStorage/cookie; фронт добавляет токен в заголовки для admin API.

## Тесты
- Unit: миграция + CRUD с role.
- API: логин admin/user; admin может /api/users, user получает 403; export events 403 для user.
- E2E: логин как admin → редактирование/удаление; логин как user → отказ.

## Почему нужно (даже в учебном)
- Показывает базовые принципы RBAC.
- Демонстрирует разграничение доступа к чувствительным операциям (управление пользователями/логами).
- Легко расширить под “оператор” и “техподдержку”.
